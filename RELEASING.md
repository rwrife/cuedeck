# Releasing CueDeck

CueDeck ships as native desktop installers built with
[**electron-builder**](https://www.electron.build/). This document covers how to
produce those artifacts locally and (optionally) via CI.

The electron-builder configuration lives in the `build` block of
[`package.json`](package.json).

---

## Prerequisites

```bash
npm install
```

Packaging downloads the Electron binary and, on first run, some platform tools
(e.g. the AppImage runtime). If you previously installed with
`ELECTRON_SKIP_BINARY_DOWNLOAD=1` (as CI does), re-run a plain `npm install`
first so the Electron binary is present.

## What gets built

| Platform | Targets | Output (in `release/`) |
| --- | --- | --- |
| **Linux** | `AppImage`, `deb` | `CueDeck-<version>.AppImage`, `cuedeck_<version>_amd64.deb` |
| **Windows** | `nsis` | `CueDeck Setup <version>.exe` |
| **macOS** | `dmg`, `zip` (x64 + arm64) | `CueDeck-<version>*.dmg`, `CueDeck-<version>*-mac.zip` |

All artifacts land in `release/`, which is **gitignored**.

## Build commands

```bash
npm run package          # build the current OS's default targets
npm run package:linux    # Linux: AppImage + deb
npm run package:win      # Windows: NSIS installer
npm run package:mac      # macOS: dmg + zip (x64 + arm64)
npm run package:dir      # unpacked app dir only (fast smoke test, no installer)
```

Each of these first runs `electron-vite build` (compiling `main`, `preload`, and
`renderer` into `out/`) and then invokes electron-builder. The packaged app loads
the built renderer/preload from inside the `asar` — the main process already
resolves prod paths via `app.isPackaged`, so no dev server is required at
runtime.

## Icons

The app icon is a committed **placeholder** at
[`build/icon.png`](build/icon.png) (512×512), generated by `npm run icons`
([`scripts/gen-placeholder-icon.mjs`](scripts/gen-placeholder-icon.mjs)).
electron-builder converts it to `.ico` (Windows) and `.icns` (macOS) and resizes
it for Linux automatically.

**Replace it before a public release** — see [`build/README.md`](build/README.md)
for how (drop in a real 1024×1024 PNG, or add native `icon.ico` / `icon.icns`).

## Cross-platform packaging caveats

electron-builder can only fully build a platform's installer on (or with tooling
for) that platform:

- **Linux** installers (AppImage/deb) build natively on Linux — this is the path
  verified in this repo/CI.
- **Windows** NSIS installers build natively on Windows. Building them on Linux
  requires **Wine** and is not exercised here.
- **macOS** `.dmg`/`.zip` must be built **on macOS** (Apple's tooling is
  required). They are configured here but **unverified** on non-mac hosts.

So: the Linux target is *configured and verified*; Windows and macOS are
*configured but must be built on their respective OSes (or in CI)*.

## Code signing & notarization (out of scope)

The configuration does **not** perform code signing or notarization:

- **macOS** apps will be unsigned/un-notarized and will trigger Gatekeeper
  warnings ("cannot be opened because the developer cannot be verified"). Real
  distribution needs an Apple Developer ID certificate plus notarization
  (`CSC_LINK` / `CSC_KEY_PASSWORD` and `notarize` config).
- **Windows** installers will be unsigned and may trip SmartScreen. Real
  distribution needs an Authenticode certificate.

These are intentionally deferred; wire in the relevant secrets/config when a
signing identity is available.

## Automated GitHub release on tag push

The GitHub Actions workflow at
[`.github/workflows/release.yml`](.github/workflows/release.yml) builds
Linux/Windows/macOS artifacts on native runners for tags like `v0.1.0`, then
automatically creates/updates the corresponding GitHub Release and uploads all
installers as release assets. Artifact versions are derived from the pushed tag
(`vX.Y.Z` → `X.Y.Z`) during CI, so installer filenames/metadata match the
release version without needing a manual `package.json` bump commit. It still
does **not** code-sign or notarize.

```bash
# cut a release build
git tag v0.1.0
git push origin v0.1.0
```
